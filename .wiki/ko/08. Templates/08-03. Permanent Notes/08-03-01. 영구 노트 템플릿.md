# 제목

## 목적

이 노트가 어떤 기능, 구조, 기준을 장기 지식으로 고정하는지 작성합니다. 다음 에이전트와 사람이 이 노트만 읽어도 판단의 출발점을 잡을 수 있어야 합니다.

## 핵심 명제

- 장기적으로 재사용할 확정 지식을 한 문장으로 작성합니다.

## 적용 범위

### 포함

- 이 지식을 적용하는 시스템, 기능, 코드 영역, 운영 상황을 작성합니다.

### 제외

- 비슷해 보이지만 이 지식을 그대로 적용하면 안 되는 영역을 작성합니다.

## 배경과 맥락

- 이 기준이 생긴 이유, 이전 방식, 현재 구조, 관련 제약을 설명합니다.
- 제품/아키텍처 맥락을 모르는 에이전트가 읽어도 이해할 수 있게 약어와 경계를 풀어 씁니다.

## 구조와 흐름

- 시스템 구성요소, 처리 흐름, 데이터 흐름, 책임 경계를 작성합니다.
- 필요하면 Mermaid 또는 text diagram을 사용합니다.

```text
입력
 -> 처리 주체
 -> 출력
```

## 실제 구현 상세

- 기능/아키텍처 영구 노트라면 package, class, method, DTO, domain model, entity, repository, external client, config, queue, table 등 실제 구현 식별자를 작성합니다.
- 표준 노트라면 실제 class 목록 대신 이 표준을 적용할 때 확인해야 하는 구조, 허용/금지 의존성, 리뷰 질문, 예외 기록 방식을 작성합니다.
- 소스 코드를 그대로 복사하지 않고, 다음 에이전트가 프로젝트를 열기 전에도 실제 구현을 읽은 것과 같은 판단을 시작할 수 있게 작성합니다.

## 상세 기준

- 구현, 설계, 운영에서 따라야 하는 구체 규칙을 작성합니다.
- 표와 목록을 사용해 값, 책임, 허용/금지 기준을 명확히 합니다.

## 예외와 금지 사항

- 예외적으로 허용되는 조건을 작성합니다.
- 금지되는 패턴과 그 이유를 작성합니다.

## 운영 및 검증 기준

- 이 지식이 올바르게 적용되었는지 확인할 수 있는 테스트, 리뷰 기준, 점검 포인트를 작성합니다.
- 코드 명령이 아니라 제품/아키텍처 검증 기준으로도 읽히게 작성합니다.
